# Scheduled Work: Tier 2 HWM `load_group` Isolation Fix

**Status:** Scheduled, not started
**Priority:** Prerequisite for B2 integration commit (`docs/planning/b2-watermark-scale-out-design.md`)
**Effort:** 1–2 days implementation + one-shot backfill
**Blocks:** B2 codegen integration into LHP; any multi-flowgroup deployment against the same `(source_system_id, schema, table)` surface
**Created:** 2026-04-24

---

## Problem

Tier 1 (shipped 2026-04-24, commit `63bcfdb`) filters HWM lookups by `source_system_id`. This closes cross-source poisoning (e.g., `pg_supabase` vs `freesql_catalog` sharing a schema+table name).

It does **not** close same-source cross-flowgroup poisoning. Two LHP flowgroups both targeting `source_system_id = 'pg_supabase'` against overlapping `(schema, table)` pairs still collide: whichever flowgroup runs first advances the HWM; the other reads that HWM as its starting point and correctly returns zero new rows on a static-data slice.

B2's flowgroup-oriented batching relies on per-pipeline HWM identity. The registry has no column that distinguishes one LHP flowgroup from another today.

Worked example of the remaining hazard (post-Tier-1):

- Flowgroup `customers_daily` loads `humanresources.employee` from `pg_supabase`, completes, registry has `watermark_value = 2014-12-26T09:17:08`.
- Flowgroup `hr_reporting_hourly` loads the same `humanresources.employee` from `pg_supabase` for a different downstream purpose. Its first-run HWM lookup returns `2014-12-26T09:17:08` (Tier 1 filters by `source_system_id`, matches). `hr_reporting_hourly` reads zero new rows on a static dataset.
- Silent, no error signal.

Tier 2 closes this by adding a `load_group` axis to the registry.

---

## `load_group` definition

`load_group` = the LHP flowgroup field's value, verbatim.

Examples from the production fleet:

- `customers_daily`
- `inventory_hourly`
- `hr_reporting_hourly`

The `flowgroup` field is already a required string on `FlowGroup`; no new schema surface. Two distinct flowgroups touching the same `(source_system_id, schema, table)` get distinct `load_group` values **by construction** — the isolation guarantee falls out of existing LHP semantics.

Environment separation (dev/qa/prod) remains handled by ADR-004's registry placement: `metadata.<env>_orchestration.watermarks`. Tier 2 does not change env scoping.

---

## Fix

Add `load_group STRING` column to the watermarks table. Extend `WatermarkManager` methods to accept `load_group: Optional[str] = None` as a kwarg (back-compat so the existing fleet keeps working while migrating per-flowgroup).

### Signature extension (all affected methods)

```python
# src/lhp_watermark/watermark_manager.py

def insert_new(
    self,
    run_id: str,
    source_system_id: str,
    schema_name: str,
    table_name: str,
    watermark_column_name: str,
    previous_watermark_value: Optional[str],
    extraction_type: str,
    *,
    load_group: Optional[str] = None,  # NEW
) -> None: ...

def get_latest_watermark(
    self,
    source_system_id: str,
    schema_name: str,
    table_name: str,
    *,
    load_group: Optional[str] = None,  # NEW
) -> Optional[str]: ...

def mark_landed(self, run_id: str, *, load_group: Optional[str] = None) -> None: ...
def mark_complete(self, run_id: str, row_count: int, new_hwm: Optional[str],
                  *, load_group: Optional[str] = None) -> None: ...
def mark_failed(self, run_id: str, error_class: str, error_message: str,
                *, load_group: Optional[str] = None) -> None: ...
def get_recoverable_landed_run(
    self,
    source_system_id: str,
    schema_name: str,
    table_name: str,
    *,
    load_group: Optional[str] = None,  # NEW
) -> Optional[str]: ...
```

### SQL filter extension (`get_latest_watermark`)

Three-way filter supports migration coexistence:

```sql
SELECT MAX(watermark_value) AS hwm
FROM <wm_catalog>.<wm_schema>.watermarks
WHERE source_system_id = :source_system_id
  AND schema_name      = :schema_name
  AND table_name       = :table_name
  AND status           = 'completed'
  AND (
        load_group = :load_group
        OR (:load_group IS NULL AND load_group IS NULL)
  )
```

This preserves pre-Tier-2 behaviour when callers pass `load_group=None` and strictly filters when callers pass a value — so a B2 flowgroup's HWM lookups are isolated from a legacy flowgroup's registry rows.

## Migration steps (safe, reversible)

1. **Add column as nullable.** The auto-DDL path (`_ensure_table_exists`) adds `load_group STRING` on next write. Existing rows have `load_group IS NULL`.

2. **Deploy `WatermarkManager` accepting `load_group: Optional[str] = None`** everywhere. `get_latest_watermark` uses the three-way filter above so mixed callers coexist.

3. **One-shot backfill.** All existing rows pre-date B2 and the manifest table, so there is no per-flowgroup remap to attempt:

   ```sql
   UPDATE <wm_catalog>.<wm_schema>.watermarks
   SET load_group = 'legacy'
   WHERE load_group IS NULL;
   ```

   Run once per environment (dev/qa/prod). Idempotent.

4. **Per-flowgroup migration.** When a flowgroup adopts B2, its first run writes `load_group = '<flowgroup_name>'`. The `'legacy'` row remains the HWM ceiling for that `(source, schema, table, 'legacy')` tuple — orphaned but harmless; a future cleanup pass can drop legacy rows once their flowgroups have successfully migrated.

5. **Do not flip to `NOT NULL` without explicit follow-up.** Flipping `load_group` to `NOT NULL` requires either (a) a Delta writer version ≥ 9 and a `DEFAULT` value, or (b) enabling `delta.columnMapping.mode = 'name'`. Option (b) is a **Delta protocol upgrade** that affects every reader of the watermarks table (Spark 3.0+ required; some external readers and older clients break on column-mapped tables). Treat it as a one-way operational decision, not a routine migration step. Test on a Delta shallow clone before enabling on the live registry. Defer the flip until Tier 2 has been live ≥ 30 days with zero NULL writes from the active fleet.

## Verification

Three behavioural checks. No unit-test harness replacement; extend existing `tests/lhp_watermark/` suite.

### V1 — Smoke: isolation works

Insert two marker rows with same `(source_system_id, schema, table)` but different `load_group`:

```sql
INSERT INTO <wm_catalog>.<wm_schema>.watermarks
  (run_id, source_system_id, schema_name, table_name, load_group,
   watermark_column_name, watermark_value, previous_watermark_value,
   row_count, extraction_type, status, created_at, updated_at)
VALUES
  ('tier2_probe_lg_a', 'pg_supabase', 'humanresources', 'employee', 'lg_a',
   'ModifiedDate', '2014-12-26T09:17:08', NULL, 100, 'full',
   'completed', current_timestamp(), current_timestamp()),
  ('tier2_probe_lg_b', 'pg_supabase', 'humanresources', 'employee', 'lg_b',
   'ModifiedDate', '2099-01-01T00:00:00', NULL, 1, 'full',
   'completed', current_timestamp(), current_timestamp());
```

Assertions:

- `wm.get_latest_watermark('pg_supabase', 'humanresources', 'employee', load_group='lg_a')` returns `'2014-12-26T09:17:08'`.
- `wm.get_latest_watermark('pg_supabase', 'humanresources', 'employee', load_group='lg_b')` returns `'2099-01-01T00:00:00'`.
- Neither leaks to the other.

Cleanup (marker-scoped, not table-scoped):

```sql
DELETE FROM <wm_catalog>.<wm_schema>.watermarks
WHERE run_id IN ('tier2_probe_lg_a', 'tier2_probe_lg_b');
```

### V2 — Regression: Tier 1 behaviour preserved

Run the full `tests/lhp_watermark/test_get_latest.py` suite unchanged. All existing assertions must pass — Tier 2 is additive, must not regress cross-source filtering.

### V3 — Legacy fallback: `load_group=None` still works

`wm.get_latest_watermark('pg_supabase', 'humanresources', 'employee', load_group=None)` returns the `'legacy'` row's watermark value (the three-way filter's `OR` branch). Proves the migration safety valve — flowgroups not yet on B2 keep running against their legacy registry rows without any code change.

## Out of scope

- Tier 3 per-integration registry tables. Deferred unless Tier 2's row-level isolation creates query-performance concerns at N ≥ 1000 flowgroups.
- Changing ADR-004's registry placement decision. Env-scoped catalogs remain the outer isolation boundary.
- Big-bang fleet migration. Per-flowgroup adoption is the intended path; the `'legacy'` row pattern supports gradual rollout.
- Flipping `load_group` to `NOT NULL`. Deferred ≥ 30 days post-Tier-2 and gated on Delta-protocol-upgrade review (see Migration step 5).
